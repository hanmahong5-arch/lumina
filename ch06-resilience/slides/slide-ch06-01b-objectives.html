Ch06 Resilience Engineering · Learning Objectives

Objectives

After this chapter you will be able to:

学完本章后你将能够：

  1. Explain why withRetry() is an AsyncGenerator (not a plain async function) and demonstrate how it yields SystemAPIErrorMessages to provide transparent retry feedback to UI / 解释 withRetry() 为何是 AsyncGenerator（而非普通 async 函数），并演示它如何通过 yield SystemAPIErrorMessages 向 UI 提供透明的重试反馈

  2. Derive the retry delay for attempt N using getRetryDelay()'s 4-tier formula (Retry-After header → exponential 500ms*2^n → 32s cap → 25% jitter) and contrast it with persistent mode's 5-minute cap / 用 getRetryDelay() 的四层公式（Retry-After header → 指数 500ms*2^n → 32s 上限 → 25% 抖动）推导第 N 次重试的延迟，并与持久模式的 5 分钟上限进行对比

  3. Trace the model fallback trigger: 3 consecutive 529 errors → FallbackTriggeredError → tombstone marking → fallbackModel switch, and justify the choice of 3 as threshold / 追踪模型降级触发链：连续 3 次 529 错误 → FallbackTriggeredError → 墓碑标记 → fallbackModel 切换，并论证选择 3 作为阈值的合理性

  4. Reconstruct a conversation from a JSONL transcript file using buildConversationChain()'s parentUuid walk, cycle detection via seen set, and recoverOrphanedParallelToolResults() post-processing / 利用 buildConversationChain() 的 parentUuid 遍历、seen 集合环检测和 recoverOrphanedParallelToolResults() 后处理，从 JSONL 转录文件重建对话

  5. Classify the 3 interrupt types detected by detectTurnInterruption() (none, interrupted_turn, interrupted_prompt) and describe the 4-layer message filter applied before session resume / 分类 detectTurnInterruption() 检测的 3 种中断类型（none、interrupted_turn、interrupted_prompt），并描述会话恢复前应用的 4 层消息过滤

Prerequisites / 前置知识

Ch01 Core Engine (query loop error handling, Terminal conditions). Understanding of HTTP status codes (429, 529, 401, 413) and exponential backoff concepts. / Ch01 核心引擎（查询循环错误处理、Terminal 条件）。理解 HTTP 状态码（429、529、401、413）和指数退避概念。
