529 前台/后台分区 / Foreground vs Background

Each retry = 3-10x amplification factor -- must be careful

Foreground (Interactive)

  • User is actively waiting
  • Max 3 retries before fallback
  • Short backoff (500ms-32s)
  • Show countdown to user
  • Triggers model fallback on failure

FOREGROUND_529_RETRY_SOURCES

Background (Batch/Task)

  • Automated task, no user staring
  • More retries allowed
  • Longer backoff acceptable
  • No fallback trigger
  • Silent retry with logging

Background task sources

Amplification Risk

HTTP 529 = "overloaded." If 1000 clients each retry 10x, the server sees 10,000 requests instead of 1,000. The foreground cap of 3 retries limits amplification to 3x. Background tasks can afford patience but must also limit to prevent thundering herd.
