Ch09 Ink UI · Learning Objectives

Objectives

After this chapter you will be able to:

学完本章后你将能够：

  1. Explain the responseLengthRef bypass pattern: why Spinner reads a React ref directly instead of subscribing to state, and how this avoids per-token re-renders during streaming / 解释 responseLengthRef 绕过模式：为何 Spinner 直接读取 React ref 而非订阅 state，以及此方式如何避免流式传输期间每个 token 触发重渲染

  2. Trace the streaming data flow from query()'s AsyncGenerator through handleMessageFromStream() to setMessages() and useDeferredValue(messages), showing how React 18 concurrent features prioritize user input over message rendering / 追踪从 query() 的 AsyncGenerator 经 handleMessageFromStream() 到 setMessages() 和 useDeferredValue(messages) 的流式数据流，展示 React 18 并发特性如何优先保证用户输入响应

  3. Identify REPL.tsx's 5 logical zones (Imports+DCE, State+Refs, Effects, Handlers, Render) and justify the "pragmatic monolith" decision against splitting into smaller files / 识别 REPL.tsx 的 5 个逻辑区域（导入+DCE、状态+Refs、Effects、Handlers、Render），并论证不拆分为更小文件的"务实巨型组件"决策

  4. Describe the 4 PromptInput modes (prompt, bash, plan, vim), the platform-conditional mode-cycle keybinding (shift+tab vs meta+m), and how 56 feature() calls within REPL.tsx enable build-target differentiation / 描述 PromptInput 的 4 种模式（prompt、bash、plan、vim）、平台条件化的模式切换快捷键（shift+tab vs meta+m），以及 REPL.tsx 中 56 处 feature() 调用如何实现构建目标差异化

  5. Compare Ink (React for Terminal) against readline, Blessed, and Bubbletea across 6 dimensions (paradigm, layout engine, state management, concurrency, ecosystem, maintenance) and justify why Ink was selected / 从 6 个维度（范式、布局引擎、状态管理、并发、生态、维护状态）对比 Ink（终端 React）与 readline、Blessed、Bubbletea，并论证选择 Ink 的合理性

Prerequisites / 前置知识

Ch01 AsyncGenerator streaming model. Working knowledge of React (hooks, reconciliation, useDeferredValue). Familiarity with terminal ANSI escape codes is helpful but not required. / Ch01 AsyncGenerator 流式模型。掌握 React 工作知识（hooks、reconciliation、useDeferredValue）。了解终端 ANSI 转义码有帮助但非必需。
