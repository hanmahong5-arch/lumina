SQLite 持久层 — StateDbHandle

为什么选 SQLite？（5 个理由）

1

零依赖部署：单文件 DB，无需独立 DB 服务；CLI 工具的理想选择

2

ACID 事务：goal 状态写入原子性，agent crash 不会产生半写状态

3

跨进程访问：多个 Codex 实例可读同一 DB（WAL 模式），Claude Code 的内存无此能力

4

可审计：goals、token 消耗、session 历史均可用标准 SQL 查询，无需自定义解析器

5

schema 迁移：rusqlite migration 支持版本化升级，markdown 文件无此能力

手动写入，无结构约束

  • 人类可读，LLM 也可读
  • 无事务保证，写入中断 = 文件损坏
  • 无跨进程锁，并发写入危险
  • 内容由用户/LLM 自由决定，无 schema 验证

权衡：Markdown 对人友好，SQLite 对程序友好。Codex 的 goal 是"程序自主执行的目标"，因此选择结构化存储；Claude Code 的 memory 是"给 LLM 读的上下文"，因此选择文本。
