场景:agent 写状态写到一半,进程崩了

这恰恰检验「6 个持久化 crate」值不值——长任务里崩溃是常态,不是意外。

Codex —— SQLite 事务(thread_goals 表)

  • 写入在一个事务里;崩溃发生在 commit 前 → 整笔回滚,旧状态完好
  • WAL 模式 + ACID:不会出现「写一半」的中间态
  • 重启后照常查询:

SELECT objective, status, tokens_used

FROM thread_goals WHERE status='Active';

✓ 状态可恢复、可审计、可继续

Claude Code —— markdown 文件

  • 状态写进 ~/.claude/memory/ 下的 .md 文件
  • 写到一半崩溃 → 文件半截损坏,无事务边界
  • 下次启动:解析报错,或静默读到脏数据,状态悄悄丢失
  • 胜在简单可读(能直接 cat),适合人维护的当前会话

✗ 崩溃边界无保障

这就是假设之差:Codex 假设 agent 会长期、跨会话、可能崩溃地自主运行,所以状态必须工业化、事务化;Claude Code 假设以当前会话为主、状态由人维护,markdown 足够。两者都对——取决于你押 agent 在哪种工作模式。
