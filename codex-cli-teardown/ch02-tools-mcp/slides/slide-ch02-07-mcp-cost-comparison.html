MCP 集成对比："同一问题，不同实现成本"

Claude Code — 依赖 npm MCP SDK

  • 省： MCP 客户端代码量少，直接 import @modelcontextprotocol/sdk
  • 省： 与核心引擎集成简单（同语言同进程，直接 import & use）
  • 缺： MCP server 子进程不自动套用 sandbox policy
  • 缺： MCP server 默认信任，permission prompt 兜底
  • 缺： 无法深度集成 network proxy（MITM 拦截 ch03 话题）

Codex — 自研 codex-mcp crate

  • 贵： 独立 crate + 跨 crate 边界 trait 实现，代码量更多
  • 得： spawn MCP 子进程时自动套 codex-linux-sandbox policy
  • 得： 与 network proxy 深度集成（MITM 能拦截 MCP 流量）
  • 得： MCP server 默认 untrusted，sandbox + MITM 双重兜底
  • 得： hot config reload，连接生命周期完全受控

结论：Codex 自研 MCP 客户端是为了 sandbox 深度集成，不是为了减少依赖。一个 npm 现成 SDK 不会"知道"在 spawn MCP server 子进程时要套上 codex-linux-sandbox——这种深度只能手写。
