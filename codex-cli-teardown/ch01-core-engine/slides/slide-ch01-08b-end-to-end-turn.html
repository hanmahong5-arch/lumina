一次完整 Turn 的端到端追踪

从你敲下 prompt 到答案返回,数据如何穿过各个 actor —— 所有跨界通信只走 Submission / Event 两条可序列化队列。

T0 · 前端

Client

tx_sub.send(Submission { Op::UserTurn }) —— 你的 prompt 入提交队列

T1 · 调度

submission_loop

rx_sub.recv() 取出提交 → 派发一个新 turn　handlers.rs:962

T2 · Turn

tokio task

client_session.stream(prompt, …) 向模型发起流式请求　turn.rs:1835

T3 · Turn

tokio task

每个 token → send_event(EventMsg::AgentMessageContentDelta) ×N　turn.rs:1554

T4 · Turn

tokio task

模型请求执行命令 → 发出 EventMsg::ExecCommandBegin

T5 · 子进程

subprocess

codex-linux-sandbox 在沙箱内执行该命令(独立 helper 二进制)

T6 · Turn

tokio task

EventMsg::ExecCommandEnd → 结果回灌模型,继续流式生成

T7 · Turn

tokio task

终答 AgentMessage → EventMsg::TurnComplete → 前端 rx_event 收尾

为什么这样设计:前端自始至终只通过 rx_event 观察,从不直接驱动引擎或触碰工具结果。把控制流压成两条可序列化队列,才让"引擎是独立二进制、前端是协议消费者"成立。
